# 1. Summary

This RFC defines the shared protocol and connection-state contract between the Karaoke Webview Host and external protocol participants such as the Android Companion.

The protocol transports client requests, capabilities, diagnostics, host projections, and microphone observations while preserving the Host as the sole authoritative owner of karaoke domain state.

This RFC defines semantic behavior, lifecycle, identity, transport responsibilities, timing, buffering, recovery, security, versioning, and diagnostics.

It intentionally does not freeze a concrete packet byte layout or implementation library.

```text
Android Companion
        │
        ▼
Platform Protocol
        │
        ▼
Network MicrophoneSource
        │
        ▼
MicrophoneChannel
        │
        ▼
CaptureSession
        │
        ├── Meter
        ├── Scoring
        └── Recording
```

---

# 2. Motivation

The Windows Host and Android Companion are independently governed implementations beneath the Karaoke Webview Platform.

A shared protocol is required so they may interoperate without either implementation redefining:

- authority;
- identity;
- microphone ownership;
- assignment;
- capture;
- recovery;
- performance state.

RFC-P-008 is the platform bridge between those implementations.

Neither side may independently redefine the protocol semantics established here.

---

# 3. Decisions

## 3.1 Authority

### RFC-P-008.1

**Decision**

The Host is the sole authority for protocol sessions and accepted microphone streams.

**Reason**

Protocol participation must not create distributed domain authority.

---

### RFC-P-008.2

**Decision**

Clients may send:

- connection requests;
- pairing requests;
- capabilities;
- diagnostics;
- lifecycle requests;
- microphone observations.

Clients may not directly commit Host-domain mutations.

**Reason**

Client actions are requests or observations. The Host decides authoritative outcomes.

---

### RFC-P-008.3

**Decision**

Singer, channel, assignment, queue, performance, score, and history data sent to clients are read-only Host projections.

**Reason**

Projection does not transfer ownership.

---

# 4. Identity Model

## RFC-P-008.4

**Decision**

The following identities are distinct:

- `ClientDeviceId`
- `ClientConnectionId`
- `ProtocolSessionId`
- `NetworkMicrophoneSourceId`
- `AudioStreamId`

**Reason**

Each identity belongs to a different lifecycle and must not be treated as interchangeable.

---

## RFC-P-008.5

**Decision**

`ClientDeviceId` represents a stable opaque installation identity generated by the client.

It must not be derived from:

- IP address;
- MAC address;
- hardware serial;
- phone model;
- user-visible device name alone.

**Reason**

Stable protocol identity should remain private, portable, and independent of network topology.

---

## RFC-P-008.6

**Decision**

Pairing identity is separate from singer identity.

**Reason**

A trusted device is not inherently a karaoke participant.

---

## RFC-P-008.7

**Decision**

Protocol Version 1 supports one active `NetworkMicrophoneSource` and one active `AudioStream` per client protocol session.

**Reason**

This matches the current one-source-per-client architecture while allowing future extension.

---

# 5. Discovery, Pairing, and Trust

## RFC-P-008.8

**Decision**

QR pairing is the primary onboarding mechanism.

**Reason**

QR pairing provides explicit operator intent and deterministic trust establishment.

---

## RFC-P-008.9

**Decision**

mDNS or DNS-SD may advertise available Hosts as a convenience.

Discovery alone does not establish trust.

**Reason**

Host visibility and Host authorization are separate concerns.

---

## RFC-P-008.10

**Decision**

Manual Host address entry is the fallback discovery mechanism.

**Reason**

Manual entry preserves usability when automatic discovery is unavailable.

---

## RFC-P-008.11

**Decision**

Initial pairing uses a short-lived, single-use Host-issued token.

**Reason**

Permanent credentials must not be exposed through reusable pairing material.

---

## RFC-P-008.12

**Decision**

Successful pairing establishes revocable trust between one Host and one client installation.

Trust is Host-specific.

**Reason**

A client trusted by one Host must not automatically be trusted by another.

---

## RFC-P-008.13

**Decision**

Trust does not imply:

- singer identity;
- microphone assignment;
- stream authorization;
- active capture.

**Reason**

These are separate platform relationships.

---

# 6. Connection State Machine

## RFC-P-008.14

**Decision**

The shared connection lifecycle includes:

```text
disconnected
→ discovering
→ connecting
→ pairing
→ negotiating
→ connected
→ reconnecting
→ closed
```

Stream authorization is a Host-controlled capability layered upon a connected session.

**Reason**

A client may remain connected without being authorized to send audio.

---

## RFC-P-008.15

**Decision**

Negotiation must complete before stream authorization.

**Reason**

Version, capability, and format compatibility must be known before audio is accepted.

---

## RFC-P-008.16

**Decision**

The Host may authorize or revoke streaming without closing the client connection.

**Reason**

Connection and stream use are independent.

---

## RFC-P-008.17

**Decision**

A `ProtocolSessionId` may survive a bounded reconnect window while the underlying connection changes.

**Reason**

Brief network interruptions should not require complete re-pairing.

---

## RFC-P-008.18

**Decision**

Closing or expiring a protocol session disconnects the corresponding network source but does not delete Host-owned channel or assignment identity.

**Reason**

Connection, source health, channel identity, and assignment are separate lifecycles.

---

# 7. Control Plane

## RFC-P-008.19

**Decision**

The control plane is:

- reliable;
- ordered;
- bidirectional;
- encrypted in production.

**Reason**

Pairing, negotiation, authorization, projection, and lifecycle messages require reliable delivery.

---

## RFC-P-008.20

**Decision**

Client-originated state changes are expressed as requests.

Host responses contain authoritative outcomes.

**Reason**

This preserves Platform Authority.

---

## RFC-P-008.21

**Decision**

Control messages use:

- explicit message type;
- protocol version;
- request identifier where relevant;
- reply or correlation identifier where relevant;
- structured payload;
- typed reason code;
- protocol session context.

**Reason**

Typed and correlated messages improve compatibility, testing, and diagnostics.

---

## RFC-P-008.22

**Decision**

The control plane may carry:

- pairing;
- trust confirmation;
- version negotiation;
- capability negotiation;
- heartbeat;
- timing exchange;
- session resume;
- stream authorization;
- stream revocation;
- capture start or stop requests;
- client diagnostics;
- immutable Host projections;
- errors;
- closure reasons.

**Reason**

These operations require reliable, ordered semantics.

---

## RFC-P-008.23

**Decision**

The initial production control transport shall use either:

- WebSocket over TLS; or
- a directly equivalent TLS-protected ordered stream.

The exact implementation library remains an implementation decision.

**Reason**

This preserves the chosen reliable encrypted control-plane architecture without coupling the RFC to one framework.

---

# 8. Audio Plane

## RFC-P-008.24

**Decision**

The audio plane uses a custom low-latency datagram transport.

**Reason**

Real-time microphone audio prioritizes bounded latency over complete delivery.

---

## RFC-P-008.25

**Decision**

Production audio packets are encrypted and authenticated.

**Reason**

Microphone audio and stream identity require privacy, integrity, and replay protection.

---

## RFC-P-008.26

**Decision**

Audio packets are never retransmitted.

**Reason**

Retransmission would deliver stale audio and increase latency.

---

## RFC-P-008.27

**Decision**

Late or stale packets are discarded and never replayed.

**Reason**

The protocol must preserve real-time behavior.

---

## RFC-P-008.28

**Decision**

Protocol Version 1 uses:

- mono audio;
- 48 kHz sample rate;
- signed PCM16;
- 10 ms frames.

**Reason**

This provides a simple, deterministic, high-quality proof-of-concept format suitable for local Wi-Fi.

---

## RFC-P-008.29

**Decision**

Every Version 1 audio packet includes sufficient information to identify and align the frame, including:

- audio protocol version;
- short negotiated `AudioStreamId`;
- sequence number;
- first sample index;
- monotonic capture timestamp;
- payload sample count;
- audio payload.

**Reason**

The Host must be able to validate, order, align, and diagnose the stream.

---

## RFC-P-008.30

**Decision**

Long-lived trust credentials and authentication tokens are not repeated in every audio packet.

**Reason**

The audio plane should use short negotiated stream identity and session-derived authentication material.

---

## RFC-P-008.31

**Decision**

The Host accepts audio only for a currently authorized `AudioStreamId`.

**Reason**

A connected client must not transmit accepted audio without Host approval.

---

# 9. Clock and Timeline Alignment

## RFC-P-008.32

**Decision**

Audio packets contain both:

- sample position;
- monotonic capture timestamp.

**Reason**

Sample position establishes stream continuity. Timestamp establishes temporal relationship to the Host clock.

---

## RFC-P-008.33

**Decision**

Clock offset is estimated using repeated Host/client timing exchanges over the control plane.

**Reason**

One timing exchange is insufficient for stable latency and drift estimation.

---

## RFC-P-008.34

**Decision**

Clock synchronization quality is explicit and observable.

Suggested quality states include:

- unsynchronized;
- estimating;
- stable;
- degraded.

**Reason**

Consumers must distinguish reliable timing from uncertain timing.

---

## RFC-P-008.35

**Decision**

Poor clock quality may affect future scoring eligibility but does not destroy source, channel, or assignment identity.

**Reason**

Timing quality and domain identity are separate concerns.

---

# 10. Buffering, Jitter, and Packet Loss

## RFC-P-008.36

**Decision**

Client and Host audio buffering is strictly bounded.

**Reason**

The protocol must never trade unbounded latency for completeness.

---

## RFC-P-008.37

**Decision**

When the client’s outgoing queue overflows, the oldest stale unsent audio frames are discarded.

**Reason**

The newest available audio best preserves real-time behavior.

---

## RFC-P-008.38

**Decision**

The Host uses a bounded jitter buffer.

Initial target buffering may be configurable within an implementation-defined range.

**Reason**

Local-network jitter must be absorbed without allowing latency to grow indefinitely.

---

## RFC-P-008.39

**Decision**

Missing packets produce explicit stream gaps in the Host’s normalized stream.

They are not recovered through retransmission.

**Reason**

Downstream consumers need an honest representation of missing audio.

---

## RFC-P-008.40

**Decision**

The following are observable diagnostics:

- packet loss;
- sequence gaps;
- jitter;
- client queue depth;
- Host jitter-buffer depth;
- locally dropped frames;
- late packet count.

**Reason**

Protocol health must be measurable.

---

# 11. Reconnect and Recovery

## RFC-P-008.41

**Decision**

Control-plane loss immediately revokes effective stream authorization.

**Reason**

Audio must not continue without an authoritative Host session.

---

## RFC-P-008.42

**Decision**

During reconnect grace, a client may keep local microphone capture alive for a short bounded period.

Captured frames during this period are discarded locally.

**Reason**

This may reduce hardware restart delay without creating stale audio replay.

---

## RFC-P-008.43

**Decision**

Audio captured while disconnected is never buffered for later transmission.

**Reason**

Post-reconnect replay would violate real-time semantics.

---

## RFC-P-008.44

**Decision**

Reconnect uses the previous `ProtocolSessionId` plus Host-verifiable resume material.

**Reason**

Session identity must not be recoverable through an unverified identifier alone.

---

## RFC-P-008.45

**Decision**

Protocol Version 1 uses a five-second reconnect grace period.

**Reason**

This supports brief local-network interruptions without leaving stale sessions alive indefinitely.

---

## RFC-P-008.46

**Decision**

Reconnect expiry:

- closes the protocol session;
- marks the `NetworkMicrophoneSource` disconnected;
- preserves `MicrophoneChannel`;
- preserves `MicrophoneAssignment`.

**Reason**

This conforms to the Platform microphone recovery model.

---

# 12. Security

## RFC-P-008.47

**Decision**

All production protocol traffic is authenticated.

**Reason**

The Host must reject unauthorized control and audio traffic.

---

## RFC-P-008.48

**Decision**

Production control traffic is encrypted.

**Reason**

Control messages may include pairing, trust, identity, and session information.

---

## RFC-P-008.49

**Decision**

Production audio traffic is encrypted and authenticated.

**Reason**

Microphone audio is sensitive and must not be altered or intercepted silently.

---

## RFC-P-008.50

**Decision**

Plaintext transport is permitted only through an explicit disposable development harness.

**Reason**

Early integration testing may require reduced complexity, but insecure transport must never become the production default.

---

## RFC-P-008.51

**Decision**

Trust records and credentials are revocable.

**Reason**

Users must be able to remove previously trusted clients or Hosts.

---

## RFC-P-008.52

**Decision**

Audio replay protection is based on negotiated session context, stream identity, and packet sequence.

**Reason**

Previously valid packets must not be accepted in a later stream or session.

---

# 13. Versioning and Capabilities

## RFC-P-008.53

**Decision**

The protocol uses explicit major and minor versions.

**Reason**

Protocol compatibility must be deterministic.

---

## RFC-P-008.54

**Decision**

Major-version incompatibility causes a clean rejection.

**Reason**

Incompatible protocol semantics must not be guessed.

---

## RFC-P-008.55

**Decision**

Clients declare capabilities before stream authorization.

Capabilities may include:

- audio formats;
- frame durations;
- clock support;
- diagnostics support;
- projection support;
- reconnect support.

**Reason**

The Host must know whether a client can satisfy the selected protocol profile.

---

## RFC-P-008.56

**Decision**

Optional capability mismatch does not prevent connection unless the Host marks that capability as required.

**Reason**

The protocol should degrade cleanly where possible.

---

## RFC-P-008.57

**Decision**

The negotiated audio profile remains immutable for the lifetime of one `AudioStreamId`.

**Reason**

Mid-stream format changes would complicate ordering, decoding, and downstream consumers.

---

## RFC-P-008.58

**Decision**

Changing audio format requires a new `AudioStreamId`.

**Reason**

Each stream identity must represent one stable negotiated format.

---

# 14. Host Projection Rules

## RFC-P-008.59

**Decision**

Clients may receive immutable projections of Host state, including:

- singer display identity;
- channel label;
- assignment status;
- Host mute state;
- stream authorization;
- upcoming participation state;
- operator messages.

**Reason**

Clients need presentation context without receiving domain authority.

---

## RFC-P-008.60

**Decision**

A client may retain projections temporarily in memory for presentation and reconnect continuity.

It must not persist them as authoritative domain truth.

**Reason**

UI continuity is permitted; distributed ownership is not.

---

## RFC-P-008.61

**Decision**

Stream authorization is independent from singer assignment.

**Reason**

A source may stream for diagnostics without an assignment, and an assignment may survive source disconnection.

---

# 15. Diagnostics

## RFC-P-008.62

**Decision**

The protocol exposes sufficient diagnostics to observe:

- connection state;
- stream authorization;
- heartbeat age;
- round-trip time;
- clock offset;
- clock quality;
- packets sent and received;
- sequence gaps;
- loss rate;
- jitter;
- client queue depth;
- Host jitter-buffer depth;
- dropped frames;
- reconnect count;
- reconnect duration;
- selected audio profile;
- source health.

**Reason**

Protocol quality must be measurable before scoring or recording depends upon it.

---

# 16. Initial Acceptance Criteria

The first integrated Platform protocol implementation is accepted when:

- an Android client pairs with the Host;
- the Host recognizes the client installation;
- negotiation completes;
- the Host registers a `NetworkMicrophoneSource`;
- the Host authorizes one `AudioStream`;
- Android sends Version 1 audio frames;
- the Host maps those frames into the existing capture pipeline;
- the existing Host RMS/peak meter responds;
- no live speaker monitoring occurs;
- control loss stops audio acceptance immediately;
- reconnect does not replay stale frames;
- reconnect preserves Host channel and assignment identity;
- a 30-minute screen-off test completes with bounded memory and latency;
- diagnostics expose packet loss, jitter, timing, and reconnect behavior;
- four concurrent Android clients pass an initial integration test;
- the architecture remains capable of supporting six to eight clients without changing identities or lifecycle semantics.

# 17. Consequences

## Benefits

- One shared protocol contract for all implementations.
- Host authority remains intact.
- Android remains a thin capture client.
- Future companion platforms can reuse the same protocol.
- Local and network microphone sources share the same Host pipeline.
- Protocol behavior is versioned, measurable, and testable.
- Channel and assignment identity survive network interruption.

## Trade-offs

- A custom audio plane requires packet framing, jitter buffering, clock alignment, and authenticated encryption.
- Both implementations must maintain protocol compatibility.
- Production security increases implementation complexity.
- PCM16 consumes more bandwidth than a compressed codec.

## Risks

- Incorrect clock alignment may reduce scoring accuracy.
- Poor jitter-buffer tuning may increase latency or packet loss.
- Custom cryptographic use must rely on established libraries rather than custom algorithms.
- Protocol evolution may require compatibility testing across versions.
- Android power-management behavior may vary by device.

# 19. Out of Scope

This RFC intentionally does not define:

- exact binary packet layout;
- exact serialization format;
- selected WebSocket or TLS library;
- selected UDP cryptographic library;
- cryptographic algorithm implementation details;
- Android `AudioRecord` implementation;
- Windows socket APIs;
- UI visual design;
- scoring algorithms;
- recording formats;
- live speaker monitoring;
- cloud relay;
- internet-wide connectivity;
- NAT traversal.

These belong to implementation RFCs, protocol supplements, or future Platform RFCs.

---

# 20. Non-Goals

This RFC does not:

- make clients authoritative;
- emulate Android phones as Windows microphones;
- support offline audio replay;
- guarantee packet delivery;
- support unbounded buffering;
- permit assignment ownership on Android;
- define scoring eligibility thresholds;
- define a compressed audio codec;
- define public internet operation;
- permit insecure production transport.

# 22. Implementation Notes

The protocol defines semantic contracts rather than one implementation language or library.

Implementations should use established networking and cryptographic libraries.

No implementation should invent custom cryptographic primitives.

The initial development harness may use plaintext only when:

- explicitly enabled;
- clearly labeled insecure;
- restricted to development;
- excluded from production builds where practical.

The Host meter should be the first integration consumer.

Scoring and recording must not be introduced until the network source path is independently validated.
